## 一、背景与目标

### 1.1 背景

OpenClaw 生态中，Agent 自诊断与故障排查一直缺乏面向非技术用户的友好方案。技术用户可以直接看日志、查配置、调试 API，但小白用户（大部分 OpenClaw 使用者）在面对报错时往往是：

- 看不懂日志含义
- 不知道严重程度
- 不敢自己修复（怕搞坏）
- 反复遇到同样问题

### 1.2 目标

构建一个 **面向小白用户、安全优先、可沉淀经验** 的 Agent 自诊断与自愈 Skill，让非技术用户也能：

1. 一眼看懂当前系统状态
2. 贴报错就知道是什么问题、多严重
3. 获得确认就绪的安全修复计划
4. 把修复经验沉淀下来，下次复用

## 二、目标用户

| 用户类型 | 特征 | 核心痛点 |
|---------|------|---------|
| 小白用户（主要） | 不会看日志，不懂终端、API、token、依赖等术语 | 出问题了不知道怎么办，害怕乱改搞坏系统 |
| 技术爱好者（次要） | 会一些命令行，但需要快速诊断和可复用的经验 | 不想每次都从头排查同一个问题 |

## 三、产品定位与原则

**定位**: 这不是一个真医生，而是一个会开药方的智能主动医生。

**核心原则**:
- 诊断优先于修复 — 不搞清楚不动手
- 安全优先于效率 — 宁可等确认，不可乱修复
- 沉淀优先于遗忘 — 每次修复都是经验积累

**竞品对标**: 无直接竞品（OpenClaw 生态原生 Skill）。对标理念：GitHub Copilot 的"聊天式诊断" + iOS 的"先问再改"授权流。

## 五、功能详细说明

### 5.1 体检模块（`doctor_check.py`）

**输入**: 目标目录路径 + 检查类型 + 输出格式

**流程**:
```
用户输入目录路径
    ↓
按检查类型分组执行（默认全选：skill/openclaw/system/project/log/tool）
    ↓
汇总 findings（fail/warn/info）
    ↓
计算健康分（100 起扣）
    ↓
输出报告（markdown 或 JSON）
```

**检查项全景**:

| 模块 | 检查内容 | 触发药方 |
|------|---------|---------|
| Skill 结构 | SKILL.md frontmatter、name 一致性、description 长度、agents/openai.yaml | RX-SKILL-001~004 |
| OpenClaw 组件 | HEARTBEAT 最近记录、MEMORY 目录大小、核心文件（SOUL/AGENTS/USER/IDENTITY/TOOLS） | RX-OPENCLAW-003/006/007 |
| 系统资源 | 磁盘空间 | RX-PERF-002 |
| 工程清单 | Git dirty worktree、package.json/requirements.txt 等 | RX-FILE-003 |
| 最近日志 | 扫描 .log/.err/.out 文件中的 error/exception 模式 | RX-LOG-001 |
| 工具链 | python3/ffmpeg/uv 是否可用 | RX-DEP-001 |

### 5.2 药方匹配模块（`prescription_match.py`）

**输入**: 用户粘贴的错误文本

**流程**:
```
用户输入错误文本
    ↓
解析药方库（references/prescriptions.md 表格）
    ↓
关键词/语义匹配评分
    ↓
按分数排序，取 top 3（阈值：绝对分≥8，且≥最高分×0.35）
    ↓
输出匹配结果或兜底报告
```

**匹配算法**:
- RX-ID 精确匹配：+100 分
- ModuleNotFoundError 专项分类：+70 分
- 症状关键词 token 匹配：长词+20，短词+10
- 用户输入 token 在药方全文命中：+4/token
- 阈值排除：绝对分 < 8 或 < 最高分×0.35

### 5.3 修复计划模块（`repair_plan.py`）

**输入**: 错误文本或明确的药方 ID

**输出**: 结构化的修复计划，包含：
- 药方 ID 与风险等级
- 风险策略描述
- 诊断摘要
- 建议修复步骤
- 影响范围
- 确认要求（L0 不需要，L1-L3 需要）
- 执行前检查清单
- 执行步骤
- 验证方式
- 回滚方式

### 5.4 病历管理

**记录**（`doctor_record.py`）:
- 时间戳、标题、状态（fixed/partial/blocked）
- 症状、诊断、执行动作、验证结果、下次建议
- 自动存储到 `.doctor/cases/` 目录

**搜索**（`case_search.py`）:
- 关键词匹配（中文/英文/符号）
- 按匹配分排序
- 近者优先（文件按时间倒序）

### 5.5 飞书路由（`feishu_route.py`）

| 用户消息 | 路由动作 |
|---------|---------|
| 体检/看看状态 | → 健康检查（L0，自动） |
| 报错了：错误内容 | → 药方匹配（L0，自动） |
| 帮我修一下：症状 | → 修复计划（L2，需确认） |
| 上次怎么处理：关键词 | → 病历搜索（L0，自动） |
| 带我上手 | → 引导模式（L0，自动） |

### 5.6 集成测试（`run_tests.py`）

11 项正式测试：validate / health / health-json / missing-path / auth-rx / yaml-rx / dep-rx / repair-plan / record / case-search / feishu-route

## 六、非功能性需求

### 6.1 性能
- 健康检查在正常项目上返回时间 < 5 秒（不含外部 API 调用）
- 外部调用（如 lark-cli）必须有 timeout（默认 5-10 秒）
- 飞书回复必须在 30 秒内返回，避免平台超时

### 6.2 安全
- L0 只读检查自动执行，不做任何写入
- L1-L3 写入必须在展示具体内容后等待用户确认
- L3 高风险操作默认不执行，只生成人工计划
- 病历自动脱敏：Token/Cookie/密码等自动替换为 [REDACTED]
- 不在飞书群中回显敏感信息

### 6.3 可维护性
- 所有脚本零第三方依赖（只用标准库）
- 药方库以 Markdown 表格形式维护，方便非技术成员修改
- 新增药方只需在 prescriptions.md 中加一行

### 6.4 可测试性
- 一键运行所有集成测试
- 每个核心功能都有独立测试用例/命令
- 药方匹配有 10+ 已知输入用例

### 正常流程

| 场景 | 期望结果 |
|------|---------|
| 对正常 Skill 包运行体检 | 返回健康报告，包含已通过检查项 |

| 输入未知错误文本匹配药方 | 返回兜底报告，不假装确定 |
| 生成修复计划 | 包含影响范围、确认要求、验证方式、回滚方案 |
| 记录病历 | 文件写入 `.doctor/cases/` |
| 搜索病历 | 返回匹配的病历条目 |
| 飞书路由 | 正确识别意图，L0 自动，L2 标记需确认 |

### 异常流程

| 场景 | 期望结果 |
|------|---------|
| 连续两次修复失败 | 熔断自动修复，生成复盘报告 |
| 病历目录不存在 | 自动创建 |
| 药方库为空/格式错误 | 返回"未找到明确药方" |

### 边界条件

| 场景 | 期望结果 |

| 药方名包含特殊字符 | regex token 解析正确处理 |
| 错误文本超长（>10000 字符） | 截断处理，只处理前 1000 字符 |
| 并发访问药方库 | 只读无锁，安全 |
| 飞书消息中混合多意图 | 按优先级匹配（修复 > 报错 > 体检 > 引导） |

## 十、FAQ

### Q: 这个 Skill 会自己修东西吗？
A: 不会。L0 只读自动执行，L1-L3 必须先问你再修。L3 高风险默认不修。

### Q: 会不会收集我的数据？
A: 不会。所有诊断只在本地执行，不上传任何日志、配置或个人信息。

### Q: 我加了新药方，需要改代码吗？
A: 不需要。在 `references/prescriptions.md` 表格中加一行即可，脚本自动读取。

### Q: 支持 Windows 吗？
A: 当前为 macOS/Linux 优化。Windows 需自行确保 python3 ≥ 3.9。

## 第十一模块：系统能力演进分析

### 过去：立项前缺少什么能力？

**能力缺口盘点**:
1. **标准化诊断流程缺失** — 出问题时只能肉眼读日志，没有统一的"体检-分诊-开药方-修复"路径
2. **安全护栏缺失** — 修复操作没有风险分级，用户（尤其是小白）容易误执行高风险命令
3. **经验沉淀机制缺失** — 每次修复都是一次性行为，下次遇到同样问题又得从头排查
4. **飞书集成缺失** — 不能在聊天中直接"看病"，要切换终端执行命令
5. **技能自检缺失** — 不知道 Skill 包本身结构是否完整、配置是否正确

**具体表现**:
- 遇到 `missing_scope` → 用户完全不知道这是什么，更不知道要重新授权
- 遇到 `ModuleNotFoundError` → 用户不知道用 `uv` 装还是 `pip` 装
- 同一个 TTS 问题，团队里三个人用三种不同方法排查

**根因**:
- OpenClaw 生态早期聚焦于 Agent 能力构建，自检/自愈能力属于"有更好，没有也行"的优先级
- 团队以技术用户为主，低估了小白用户的诊断需求
- 缺乏统一的"经验可复用"框架

### 现在：补了什么？闭环了吗？

**补足方案**:

| 补足模块 | 能力闭环度 | 能力半径（能做/不能做） |
|---------|-----------|----------------------|
| 只读健康检查 | ✅ 闭环 | 能做：Skill 结构校验、OpenClaw 组件检查、系统资源检查、日志扫描、工具链检查 |
| | | 不能做：网络服务可用性检查（除飞书外）、跨主机诊断 |
| 药方匹配 | ✅ 闭环 | 能做：73+ 条药方匹配，含 OpenClaw 本地高频问题 |
| | | 不能做：非结构化错误诊断（如 AI 幻觉导致的逻辑错误） |
| 修复计划 | ✅ 闭环 | 能做：生成含确认要求/验证方式/回滚方案的修复计划 |
| | | 不能做：自动化执行（安全策略不允许） |
| 病历管理 | ✅ 闭环 | 能做：记录、搜索、按匹配分排序 |
| | | 不能做：跨用户病历共享、自动标记重复病历 |
| 飞书路由 | ✅ 闭环 | 能做：识别 5 种意图并路由到本地动作 |
| | | 不能做：直接回复飞书消息（需要集成层配合） |
| 安全策略 | ✅ 闭环 | 能做：L0-L3 四级安全确认 |
| | | 不能做：动态风险评估（当前为静态分级） |
| 一键测试 | ✅ 闭环 | 能做：11 项自动化集成测试 |
| | | 不能做：大规模药方覆盖率测试（需手动补充） |

**⚡ 关键闭环评估**:
- 用户核心链路（报错 → 诊断 → 获取修复计划 → 安全修复 → 记录病历）已完全闭环
- 飞书集成链路（收到消息 → 识别意图 → 执行动作 → 返回结果）已闭环，但需要外部集成层配合推送消息

### 未来：新挑战来了怎么办？

**新挑战**:
1. **药方库膨胀** — 超过 500 条药方时，纯文本匹配可能变慢
2. **多用户病历共享** — 团队协作需要共享病历库
3. **持续守护进程** — 需要后台监控能力
4. **AI 驱动诊断** — 结合大模型提高非结构化错误的诊断准确率

**架构应对能力评估**:
- 药方库膨胀 ⏩ 匹配算法当前为 O(n)，500 条以内可接受；超过后建议引入简单索引（按首字母/场景分类预索引）
- 多用户共享 ⏩ 当前病历为本地文件，共享需要迁移到云端数据库
- 守护进程 ⏩ 需要新增 daemon 模块和 cron 集成
- AI 诊断 ⏩ 需要新增 AI 诊断模块（调用大模型 API）

**架构评级**: **B 级**（部分耦合，需评估改造成本）
- 优势：模块化程度高（每模块独立脚本），药方库纯数据驱动，添加新药方无需改代码
- 短板：病历存储为文件系统（不便于共享），匹配算法无索引（大规模性能堪忧），飞书路由缺发送层
- B→A 路径：病历迁移到 SQLite + 匹配引入索引 + 新增飞书发送模块

**架构评级定义**:
- A 级：配置驱动 + 函数内聚 + 测试完备 → 非推翻型，可扩展
- B 级：部分耦合 → 需评估改造成本
- C 级：高耦合/难以测试 → 建议推翻重构
